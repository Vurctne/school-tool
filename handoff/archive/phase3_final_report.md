## PR block

**Title:** Unify highlight-colour derivation through `toolkit.fills`; add drift guards

**Why:** Today every highlight-colour *value* in the codebase happens to agree, but only the Python→CSS token sync is actually enforced by a test. Five user-facing instruction strings and eight xlsx/Excel fill sites hard-coded their hex or BGR integer rather than deriving from `toolkit.tokens`. A future edit to `tokens.py` would silently drift the instruction text and the xlsx output without failing any existing test. This PR kills both the structural fragility and the observability gap.

**What changed:**

1. **New `toolkit/fills.py`** exporting `argb(hex)` (adds the `"FF"` ARGB alpha prefix openpyxl needs) and `bgr_int(hex)` (encodes a 6-char RGB to the Win32 COM `.Interior.Color` integer). Module docstring maps each `HL_*` constant to its `--hl-*` CSS property and flags `HL_EDITED` as a legend-only token with no active fill site.
2. **All 9 xlsx/Excel fill sites now derive from tokens.** Two openpyxl `PatternFill(fgColor=...)` calls in `master_budget/logic.py`, one in `sub_program/logic.py`, and six Win32 COM `Interior.Color` assignments in `master_budget/logic.py` all now go through the helpers.
3. **All 5 user-facing hex literals interpolate from `HL_*`.** `_HELP_TEXT` in both `sub_program/frame.py` and `master_budget/frame.py` is now an f-string; rendered output is byte-identical to before.
4. **Three drift guards in `tests/test_tokens_drift.py`.** The original CSS-sync test, plus a new AST scan that asserts no rogue hex literal survives in production string constants and a new AST scan that asserts every `PatternFill` derives its `fgColor` from `argb(<HL_*>)` or a canonical ARGB literal. A sentinel failure mode prevents the check from silently going inert if fills are later removed wholesale.
5. **Existing Win32 regression test migrated, not dropped.** `test_com_interior_colour_matches_hl_mismatch` was added 2026-04-24 after the M3 code review to pin the BGR integers. Because the refactor removes those literals, the test body now checks *(a)* that `bgr_int(HL_MISMATCH)`/`bgr_int(HL_SOURCE_ONLY)` produce the historical integers, and *(b)* that the function source contains the helper calls (and does NOT contain the bare integers). Same regression guarantee, decoupled from the refactor's surface changes.

**How tested:**
- `pytest tests/ tools/sub_program/tests/ tools/master_budget/tests/` → 191 passed, 15 skipped (environment-only), 0 failures.
- Manual verification that `argb(HL_MISMATCH) == "FFF4CCCC"`, `bgr_int(HL_MISMATCH) == 13421812`, `bgr_int(HL_SOURCE_ONLY) == 14282978`, and `_OVER_FILL.fgColor.value` / `_HIGHLIGHT_FILL.fgColor.value` / `_EXTRA_FILL.fgColor.value` all produce the expected ARGB strings at runtime.
- Manual verification that rendered `_HELP_TEXT` in both tools contains `F4CCCC` (and for master_budget also `E2F0D9`, `FFF2CC`) at exactly the same counts as before the refactor.

**Risk:**
- **Low.** The openpyxl and Win32 code paths produce identical output bytes (same ARGB, same BGR integers). Existing regression tests that previously pinned literal values by source-scan have been migrated to equivalent runtime/wiring checks — none were dropped.
- `HL_EDITED` is explicitly flagged as a ghost constant in `toolkit/fills.py`; no code path changes around it.
- Import-order changes in `master_budget/logic.py` drop an unused `HL_EDITED` import and remove the `# noqa: F401` it carried. If a downstream plugin imports `HL_EDITED` from this module (it shouldn't — it should import from `toolkit.tokens` directly), it will break; no such consumers found in the repo.

**Follow-up (not in this PR):**
- Consider extending `scripts/port_tokens.py` to emit the `HL_*` → `--hl-*` linking comments directly into `toolkit/tokens.py` so the cross-reference lives in the auto-generated file itself.
- The existing `_OVER_BG = "#" + HL_MISMATCH` in `sub_program/frame.py:91` is a Tkinter colour (different encoding from xlsx); could grow a `toolkit.fills.tk_hex()` helper if a second Tkinter site appears, but not worth it for one call site today.

## Environmental note (not a code issue)

During Phase 3 verification, the Linux bash sandbox briefly presented truncated views of `tools/sub_program/frame.py` and `tools/master_budget/frame.py` — the Windows-side view (through the Read/Edit tools) had the full files, while the Linux mount had them cut mid-statement. This caused the first run of the new AST-scan drift tests to report `SyntaxError` on those two files even though the files parsed fine from a direct `python3` import (Python was silently falling back to the `.pyc` bytecode cache). Re-writing both files from the Linux side with `pathlib.Path.write_text(...)` restored the intact content and all tests passed. This was a Cowork-sandbox filesystem sync quirk, not a code defect; no production change resulted from the repair step.
