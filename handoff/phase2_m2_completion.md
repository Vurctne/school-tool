## Reviewer-2's concerns (non-blocking; deferred)

These are real but small. Document for Phase 3 / M4 attention:

1. **`backend/src/routes/invoices.ts:43-54`** — `findSchoolById` returns 403 for a school that doesn't exist (information leak about school IDs). Functionally correct, but a 404 would be cleaner. Phase 3 fix: unify the existence + ownership check into a single `user_schools` JOIN.
2. **`backend/src/routes/invoices.ts:65`** — Invoice number sequence `count(*) + 1` is non-atomic; concurrent calls in the same second can collide on the UNIQUE `invoices.number` constraint, surfacing as an opaque 500. **Acceptable for Phase 2 launch traffic** (single-user-per-school, low concurrency), but Phase 3 should add retry-on-UNIQUE or a proper sequence helper.
3. **`backend/src/routes/licences.ts:50-58`** — Multi-school user gets the FIRST school's active licence (iteration order is `user_schools.created_at ASC`). Single-school users (the v2.0.0 launch case) are unaffected. Phase 3 should document this in a comment OR take `school_id` in the request body for explicit selection.
4. **`backend/src/routes/pos.ts:90-97`** — `GET /:id` returns 403 when `purchase_order.invoice_id` is null. The column is nullable but POSTs always set it; null is unreachable in practice. 422 would be more accurate semantically.
5. **`backend/tests/mailer.test.ts:113-122`** — HTML injection test only asserts `<` / `>` escaping; the `&#39;` (apostrophe) escape isn't asserted even though the implementation handles it. Add the assertion.
6. **`backend/tests/{invoices,pos,licences}.test.ts`** — All three duplicate the same `MIGRATION_SQL` block. Maintenance hazard. A shared `tests/setupDb.ts` helper would consolidate.

These are noted in `ORCHESTRATION-STATUS.md` "Pending decisions awaiting Ivan" only at the level of "Phase 3 polish queue" — they don't block Phase 2 ship.

```bash

wrangler secret put RESEND_API_KEY                        # from resend.com dashboard

# 3. Provision D1 + R2 (one-time)
wrangler d1 create sft
# Copy database_id into wrangler.toml under [[d1_databases]]
wrangler r2 bucket create sft-assets
wrangler queues create sft-ocr      # consumer enabled in M5
wrangler d1 execute sft --remote --file ./migrations/0001_init.sql

wrangler deploy

# 5. After deploy succeeds, copy the workers.dev URL it prints (e.g. sft-api.<account>.workers.dev)
```
