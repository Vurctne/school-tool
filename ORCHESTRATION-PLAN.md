## B. Sub-agent role definitions

All sub-agents run **Sonnet 4.6** via `Agent` tool with `model: "sonnet"`. Each role has a fixed prompt template in `_agent-prompts/` (Step 4 of the reorg).

### B.1 Discovery agent

- **Purpose:** read-only investigation of existing code, external docs, or sample files; produces a structured markdown report.
- **Input contract:** orchestrator supplies (a) a precise question, (b) the file paths or URLs to consult, (c) the report schema (table headers / fixed sections).
- **Output contract:** structured markdown only. No prose-only sections; everything in tables or itemised bullets. No edits, no recommendations beyond what the orchestrator's question asks for.
- **Boundaries (越权):** must not write any file; must not call any tool that mutates state (no MCP write actions, no `Edit` / `Write`); must not run `pytest` / build commands.
- **Failure handling:** if the agent can't reach a file (e.g. missing on disk), it returns a "file not found" line in the report — does NOT search for alternatives unless the orchestrator's prompt explicitly says to.
- **Retry rule:** orchestrator re-dispatches once with a stricter brief. After 2 failures, **🚨 escalate**.

### B.2 Implementer agent

- **Purpose:** write new code in well-scoped files. Produces a unified diff + the changed file contents.
- **Input contract:** orchestrator supplies (a) the goal, (b) the exact list of files to create/modify, (c) the API contract or test the code must satisfy, (d) the relevant `CONTEXT-SUMMARY.md` excerpt.
- **Output contract:** unified diff in `git diff --no-color` format; list of files modified; verification command (e.g. one Python invocation that exercises the new code) and its output.
- **Boundaries:** must not touch any file outside the orchestrator's list; must not edit tests unless tests are explicitly listed; must not modify `docs/`, `design_system/`, or `toolkit/tokens.py` without explicit permission.
- **Failure handling:** if the agent realises it needs a file outside scope, it returns a "scope expansion request" instead of editing — orchestrator decides.
- **Retry rule:** failed twice on same scope → **🚨 escalate**.

### B.3 Refactor agent

- **Purpose:** modify existing code while preserving behaviour. Produces diff + must-still-pass test list.
- **Input contract:** as Implementer, plus a list of tests that must still pass after the refactor.
- **Output contract:** diff + verification that listed tests still pass.
- **Boundaries:** stricter than Implementer — must not change function signatures unless explicitly authorised; must not introduce new dependencies.
- **Retry rule:** failed twice → **🚨 escalate**.

### B.4 Tester agent

- **Purpose:** write new tests OR run existing tests and report. Two modes per task.
- **Input contract:**
  - *Write mode:* orchestrator supplies the assertion to encode + the file to put it in.
  - *Run mode:* orchestrator supplies the pytest/vitest command + the expected pass/fail criterion.
- **Output contract:** new test file diff (write mode) or full test runner output (run mode).
- **Boundaries:** in run mode, **must not attempt to fix any failure** — returns the failure log only. In write mode, must not modify production code.
- **Retry rule:** orchestrator re-dispatches once if the test is structurally unsound. After 2 failures, **🚨 escalate**.

### B.5 Code-reviewer agent

- **Purpose:** independent post-implementation review with **no shared context** — different conversation thread from the implementer to catch confirmation bias.
- **Input contract:** orchestrator supplies the diff + the original goal + the acceptance criteria. Does NOT supply the implementer's reasoning.
- **Output contract:** structured markdown — `Blockers` (list, must be empty for phase to advance) / `Concerns` (list, ack-and-defer allowed) / `Suggestions` (list, optional).
- **Boundaries:** must not edit any file. Must not run code (just read).
- **Retry rule:** N/A — review output is advisory; orchestrator decides whether to dispatch a Refactor agent against the blockers.

### B.6 Fixer agent (last resort, rare)

- **Purpose:** when an Implementer / Tester fails twice on the same scope, a Fixer is dispatched with a narrower brief that includes the failure logs.
- **Input contract:** orchestrator supplies (a) failure logs verbatim, (b) the offending file, (c) what 'fixed' looks like.
- **Output contract:** diff + tests pass evidence.
- **Boundaries:** strict — only the failing file, only the failing test scope.
- **Retry rule:** Fixer failure → **🚨 escalate** (no further auto-retry; this is a planning gap).

## C. Batch & concurrency strategy

### C.1 Concurrency caps

- **Default:** ≤3 concurrent sub-agents.
- **Read-only Discovery agents:** up to 4 concurrent (cheaper, no write conflicts).
- **Implementer / Refactor / Tester (write mode):** ≤3 concurrent AND **must touch disjoint file sets**. Orchestrator pre-checks file overlap before dispatch; any overlap → serialise.
- **Code-reviewer:** always 1 at a time, after implementers finish.
- **Test-runner:** always serial (avoids fixture races).

### C.2 Inter-call pacing

- Local-only work (file reads, code generation): **no pacing required** (no rate limits to honour).
- External fetches (only relevant in Phase 4 OCR work and Phase 5 store-submission discovery):
  - **≥200 ms between calls** to the same domain.
  - **429/5xx retry:** 30 s back-off, max 3 retries, then log the failure to `ORCHESTRATION-STATUS.md §Failures` and **🚨 escalate**.

### C.3 Batch sizing

- Implementer batches: 1 file per agent (no batching). Avoids merge complexity.
- Tester batches: up to 6 assertions per test-write task (one file).
- Discovery batches: up to 10 file paths per agent.
- Code-reviewer: phase-scoped (full diff of the phase).

## D. QA cadence

### D.1 Mid-phase

- After every Implementer / Refactor agent returns: orchestrator reads the diff **before** dispatching the next agent if there's any file overlap. For disjoint file sets, can stack (within the concurrency cap).
- Spot-check rate: orchestrator reads at least every 3rd diff fully even when no overlap — guards against drift.
- If 2 consecutive agents return with scope expansion requests, orchestrator pauses and re-reads the phase plan — likely indicates a planning gap, **🚨 escalate**.

### D.2 End of phase

- 100% review: every diff in the phase, every test that was added or modified.
- Code-reviewer agent dispatched (Section B.5) with the consolidated diff.
- Test-runner agent runs the full relevant test suite on a clean checkout (no orchestrator-side state).
- Phase handoff written → `handoff/phase<N>_<topic>.md`.
- Orchestrator pauses and surfaces summary to Ivan; **does not dispatch the next phase until Ivan signs off.**

### D.3 Per-commit

- Every commit is preceded by a green test run (the relevant subset for the change).
- Commit message includes the agent that produced the change (or "orchestrator" if hand-edited).
- `ORCHESTRATION-STATUS.md` updated **on the same commit** that introduces the work — prevents status drift.

## E. Escalation mechanism

Marker: **`🚨 ORCHESTRATOR ESCALATION:`** at the start of the message.

Trigger conditions:

1. **Decision ambiguity** — spec underspecified or two reasonable readings exist.
2. **Sub-agent repeated failure** — Implementer / Refactor / Tester fails twice on the same scope, OR a Fixer fails once.
3. **Plan-vs-reality drift** — discovery surfaces material that contradicts the inventory (e.g. tools/ contains undocumented work; backend has routes already started; samples missing).
4. **Out-of-prompt judgement** — anything requiring product, legal, financial, or branding decisions (e.g. PDF renderer choice, ABN policy, store listing copy).
5. **Architecture-locked surface** — sub-agent or refactor needs to touch one of the locked surfaces in ADR-0014. Halts; Ivan ratifies as a deliberate design change.
6. **External blocker** — secret missing (Resend API key, Cloudflare token), sample data missing (Camps), Ivan-only action required (Partner Center submission, domain registration).
7. **Test failures unreproducible** — Test-runner reports a failure that orchestrator can't reproduce on a separate dispatch. Indicates flakiness or environment drift.

Format of escalation:

```
🚨 ORCHESTRATOR ESCALATION: <one-line topic>

What happened: <2-3 lines>
Decision needed: <specific question, with 2-3 options if applicable>
What I'm doing in the meantime: <continue / pause / reverted>
Files affected: <list>
```
